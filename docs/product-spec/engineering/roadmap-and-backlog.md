# 開發路線與工作拆解

## 1. 交付策略

產品建議分四個階段建立：

1. 建立可靠的核心使用循環
2. 加上情感與回饋差異化
3. 加上留存支架
4. 建立可持續更新的內容營運能力

## 2. 里程碑

| 里程碑 | 目標 | 驗收標準 |
| --- | --- | --- |
| M1 核心循環 | 使用者可完成 session 並拿到回饋 | 計時器、任務綁定、結算回饋 |
| M2 角色層 | 產品具有情感差異化 | 角色可見、休息互動、Bond 成長 |
| M3 留存層 | 產品具備首週目標結構 | 任務、成就、統計、解鎖畫面 |
| M4 內容營運層 | 團隊可持續上新 | 資料驅動內容流程成形 |

## 5. 完成定義

### 功能層

- 使用者可見流程已完成
- 驗收規則已涵蓋
- 分析事件已送出
- 持久化已驗證
- 明顯邊界情況已處理

### 內容層

- 內容已存在於資料定義中
- 即使目前只有單語，也需保留在地化路徑
- QA 可強制觸發內容做驗證

## 6. 開放問題

- MVP 任務資料是否只保留本地端
- 經濟回饋是否應設定最小支援 session 長度
- Bond 進度應以數值顯示，還是只顯示等級 / 狀態
- 在驗證留存前，房間自訂要做到多深才合理
